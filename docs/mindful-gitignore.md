# A mindful .gitignore

A `.gitignore` is a list of patterns; files matching them are invisible to git —
they never show up in `git status`, never get committed, never get pushed. Copying
a template is a fine start, but the template can't know *your* project. This page
is how to think about it.

## Three things .gitignore does NOT do

1. **It does not remove files git already tracks.** If `patients.csv` was committed
   yesterday and you add `*.csv` today, git keeps tracking it. You must
   `git rm --cached patients.csv` (removes from git, keeps the file on disk) —
   and the old versions are still in history.
2. **It does not protect against force-adds.** `git add -f` bypasses it.
3. **It does not make a folder safe for PHI.** It's an accident guard, not a
   security control. Data should live outside the repo folder when possible.

## Think in five categories

Write your .gitignore top-down in these groups, with a comment header for each.
When a future teammate opens the file, they should understand your *intent*.

### 1. Patient data & clinical files — the reason we're here

Everything that could carry PHI. For our work that means:

```gitignore
# --- Patient data: NEVER committed ---
*.dcm
*.DCM
*.dicom
*.ima
*.nii
*.nii.gz
*.mha
*.mhd
*.nrrd
*.raw
data/
exports/
```

Two strategies, pick one deliberately:

- **Blocklist** (above): name every dangerous extension. Simple, but a new export
  format slips through until you add it.
- **Quarantine folder / allowlist**: keep ALL data in one folder and ignore the
  folder itself (`data/`). Even stronger — inside such a folder nothing escapes,
  whatever its extension. The best setups do both.

For a pure-data folder you can even invert the logic — ignore everything, allowlist
code — but that's harder to reason about; only use it if you understand the
re-include (`!pattern`) rules.

### 2. Secrets & credentials

Tokens, connection strings, anything with a password. These aren't PHI but leak
access to systems that hold PHI (ARIA!).

```gitignore
# --- Secrets ---
.env
*.env
appsettings.*.json   # if you keep connection strings there — see note below
secrets.json
*.pfx
*.key
```

Note: don't blanket-ignore config files your app needs to build. Instead, commit an
`appsettings.example.json` with placeholder values and ignore the real one.

### 3. Build outputs & environments

Machine-generated, regenerated on every build. Committing them bloats the repo and
creates merge conflicts. For our stacks:

```gitignore
# --- C# / Visual Studio ---
bin/
obj/
.vs/
*.user

# --- Python ---
__pycache__/
*.pyc
.venv/
venv/
.ipynb_checkpoints/
```

### 4. Big binary artifacts

Model weights, datasets, videos. Git handles them badly (every version stored
forever). Keep them on the share or use a release/artifact mechanism.

```gitignore
*.pth
*.onnx
*.h5
*.zip
```

Exception: small static resources the code needs (an icon, a small lookup table)
belong in the repo. "Big" starts around a few MB.

### 5. OS / editor noise

```gitignore
Thumbs.db
Desktop.ini
.DS_Store
*.swp
```

## The mindful part: interrogate your own folder

Before committing, look at what's actually in the project and ask of each file type:

| Question | If yes → |
|---|---|
| Could this contain patient identifiers (even in a header I can't see)? | Category 1, or better: move it out |
| Would I be uncomfortable if this appeared on the internet? | Categories 1–2 |
| Can this be regenerated by building/running the code? | Category 3 |
| Is it larger than a few MB? | Category 4 |
| Did I create it, or did my OS/editor? | Category 5 |
| None of the above — it's code, docs, or config I wrote | **Commit it.** |

The scanner ([tools/phi_scan.py](../tools/phi_scan.py)) automates the first
question and can append what it finds:

```bash
python tools/phi_scan.py "C:\path\to\project" --update-gitignore
```

## Verifying your .gitignore actually works

```bash
git status --ignored        # ignored files should appear under "Ignored files"
git check-ignore -v file    # shows exactly which rule matched (or nothing = NOT ignored)
git ls-files                # the complete list of what git currently tracks
```

The last one is the ground truth. If a data file appears in `git ls-files`, the
.gitignore isn't protecting it — it's already tracked.

The PHI scanners check this for you too: both the Python script and the desktop
app read your .gitignore and mark findings that are already covered ("✓ ignored"
in the app, `[already in .gitignore]` in the script). Covered findings don't
fail the scan — which also means the pre-commit hook won't block commits over
data that's properly quarantined in an ignored folder.
